# 产品与数据管线

## 产品定位（对外怎么说）

见 [product-intro.md](../product-intro.md)：**独立公交查询工具**，不是官方、不是游记。

三句话版：

1. 路边查下一班 + 多少钱  
2. 数据全部来自官方 PDF，页脚写出处  
3. 日中英三语，站名以日文为准（因为站牌是日文）

---

## 功能是怎么「长」出来的

不是一次设计完，而是**场景驱动迭代**：

| 用户场景 | 功能 | 决策 |
|----------|------|------|
| 港口等船转公交 | 下一班高亮 | 默认筛「当前时间之后」 |
| 只看某条线 | 常用区间快捷 | 宫之浦港↔安房等 |
| 手机打不开 PDF | 地图页图片预览 | [笔记](../notes/2026-05-21-map-pdf-mobile-preview.md) |
| 行前查船 | `/access/` 表格 | 不做复杂筛选 |
| 想知道能不能网购船票 | booking 卡片 | [笔记](../notes/2026-05-21-access-booking.md) |

**vibe coding 技巧**：每次只加一个场景，加完**你自己用手机点一遍**再进入下一个。

## 数据管线详解

### 1. 源（Source）

- `sources/manifest.json` — 公交 PDF URL、文件名、有效期  
- `sources/access-manifest.json` — 船/巴士  
- `assets/pdf/` — 下载到本地的 PDF（构建用）

### 2. 构建（Build）

```bash
python3 scripts/build_all.py          # 全部
python3 scripts/build_all.py --map    # 只运价
python3 scripts/build_all.py --access # 只上岛
```

脚本会：

- 解析 PDF 表格 → 结构化 JSON  
- 合并 `sources/overrides/` 手工修正  
- **运价锚点校验**（11 组固定票价，错就 exit 1，防止 silent bug）  
- 输出 `data.js` 等供浏览器 `<script>` 引用  

### 3. 运行（Runtime）

浏览器加载 JS → 内存里筛选班次 → 纯前端，**无 API 请求**（除 GA4）。

## 官方 PDF 何时更新

屋久岛公交通常 **3 月、10 月** 前后改表（配合登山季）。  
流程：

1. 看 [yakukan.jp/trans](https://yakukan.jp/trans/)  
2. 下载新 PDF → 更新 manifest 里的 URL/文件名  
3. `python3 scripts/build_all.py`  
4. 人工 spot check 3～5 班车  
5. `git push` → 自动上线  

**不要**等用户报错再更新；可在 about 页写「数据核对日期」。

## 三语怎么做

- UI 字符串：JS 里 i18n 对象或独立键  
- **站名**：显示日文，中/英作副标题（现场站牌是日文）  
- 英文时刻表：单独英文 PDF 解析  

常见坑：AI 直接机翻站名 → 你要坚持「以官方 PDF 为准」。

## 可靠度设计

| 机制 | 目的 |
|------|------|
| 页脚数据来源 + 有效期 | 信任 |
| 运价「目安」标注 | 锚点估算 ≠ 官方逐站价 |
| `about` 免责声明 | 非官方、以车内为准 |
| `build_all` 校验失败即退出 | 防止错价上线 |
